<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| page DOM → capture | Arbitrary page input values are read; D-03 exclusion is the privacy gate at this boundary |
| SW snapshot → page DOM (restore) | Restored values are written back into form fields; URL-match (SW, 04-02) gates which page receives a snapshot |
| content script runs on `<all_urls>` | Capture/restore run on every site; must never persist or inject sensitive fields |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Information Disclosure | Sensitive field persisted to disk | mitigate | shouldCapture enforces D-03 at CAPTURE time (type password/hidden/file + autocomplete cc-*/one-time-code/new-password rejected) — secrets never enter the SAVE_STATE payload, so they never reach IDB. Verified by behavior test, not just restore-side filtering. |
| T-04-09 | Information Disclosure | contenteditable / shadow-DOM text leak | mitigate | Capture is restricted to `input, textarea, select` (WATCHED_SELECTORS); querySelectorAll does not pierce shadow roots and does not match contenteditable — both silently excluded by design. |
| T-04-10 | Tampering | Stale snapshot injected into wrong page | mitigate | Defense-in-depth: SW URL-match + delete-after-restore (04-02) is primary; content script only restores what GET_STATE returns for its own tab, and the SW already verified the URL. |
| T-04-11 | Denial of Service | Giant textarea / many fields bloating capture | mitigate | MAX_FIELDS (50) caps field count; MAX_FIELD_VALUE_LEN (10000) truncates each value at capture time. |
| T-04-12 | Tampering | scrollRestoration globally disabled | mitigate | history.scrollRestoration='manual' set ONLY inside startRestore (when a snapshot exists), never on normal loads (Pitfall 7) — native back/forward scroll restoration preserved for non-restore navigations. |
</threat_model>
