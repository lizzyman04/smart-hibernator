<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| content script → SW (onMessage) | Page-originated messages cross into the trusted SW; sender.tab.id is the only trustworthy identity |
| SW → IDB | SW is sole writer of tab-state (Phase 3 invariant); content script never opens IDB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Spoofing | SAVE_STATE / GET_STATE handlers | mitigate | Both handlers guard on `sender.tab?.id`; only messages from a real content-script tab are processed. tabId is derived from `sender.tab.id`, never from message body — a page cannot claim another tab's id. |
| T-04-05 | Tampering | Stale-state injection into wrong page after tabId reuse | mitigate | GET_STATE compares stored `snapshot.url` to `message.url` (D-06); mismatch returns null. delete-after-restore consumes the snapshot immediately so a coincidental URL match cannot be replayed. |
| T-04-06 | Information Disclosure | snapshot returned to caller | accept | sendResponse goes only to the originating tab's content script (Chrome routes by sender); response cannot be observed cross-tab. Data never leaves device (NFR-04). |
| T-04-07 | Denial of Service | onMessage channel left open | mitigate | GET_STATE returns `true` only on its own branch and always calls sendResponse on both resolve and catch — channel never leaks open. Other handlers return synchronously. |
</threat_model>
